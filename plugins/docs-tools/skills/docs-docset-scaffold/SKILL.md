# Doc Set Scaffold Step

Step skill for the docs-orchestrator pipeline. Follows the step skill contract: **parse args → scaffold project → write manifest**.

## Arguments

- `$1` — Workflow ID (required)
- `--base-path <path>` — Base output path (e.g., `.claude/docs/my-product-guide`)
- `--format mkdocs` — Output format (only `mkdocs` supported for now)
- `--draft` — If set, scaffold into `<base-path>/scaffold/` staging area. Otherwise, scaffold into the repo working directory.

## Output

```
<base-path>/scaffold/scaffold-info.md
```

## Execution

### 1. Parse arguments

Extract the workflow ID, `--base-path`, `--format`, and `--draft` flag from the args string.

### 2. Read the plan

Read the documentation plan from `<base-path>/planning/plan.md`.

Extract:
- **Categories**: Top-level JTBD categories (e.g., "Get started", "Configure", "Troubleshoot")
- **Parent topics**: Within each category, the parent topic titles
- **Module list**: Individual modules with their types (CONCEPT, PROCEDURE, REFERENCE) and titles
- **Project name**: From the plan's `**Project**` field or the workflow ID

If the plan does not use JTBD categories (e.g., `--no-jtbd` was used upstream), extract the top-level organizational groupings and parent topics as they appear in the plan.

### 3. Determine output location

- **Draft mode** (`--draft`): Scaffold into `<base-path>/scaffold/site/`
- **Non-draft mode**: Scaffold into the repo working directory (current directory)

### 4. Create MkDocs project structure

Generate the following files and directories:

#### `mkdocs.yml`

```yaml
site_name: <project-name>
theme:
  name: material
  features:
    - navigation.sections
    - navigation.expand
    - navigation.indexes
    - search.suggest
    - content.code.copy

markdown_extensions:
  - admonition
  - pymdownx.details
  - pymdownx.superfences
  - pymdownx.tabbed:
      alternate_style: true
  - tables
  - attr_list
  - toc:
      permalink: true

nav:
  - Home: index.md
  - <Category 1>:
    - <category-1>/index.md
    - <Parent Topic 1>: <category-1>/<parent-topic-1>.md
    - <Parent Topic 2>: <category-1>/<parent-topic-2>.md
  - <Category 2>:
    - <category-2>/index.md
    - ...
```

Populate the `nav` section from the plan's categories and parent topics. Use kebab-case for directory and file names derived from titles.

#### `docs/index.md`

```markdown
---
title: <project-name>
description: <brief description derived from plan>
---

# <project-name>

<2-3 sentence overview derived from the plan's summary or JTBD statement>
```

#### Category directories and index pages

For each category, create:
- `docs/<category>/` directory
- `docs/<category>/index.md` with:

```markdown
---
title: <Category name>
description: <brief description of the category's scope>
---

# <Category name>

<1-2 sentence overview of what this category covers>
```

#### Placeholder module files

For each parent topic listed in the plan, create a placeholder file at the expected path so that the nav references resolve. These are minimal files that the docs-writer will overwrite with full content:

```markdown
---
title: <Parent topic title>
description: <placeholder>
---

# <Parent topic title>

<!-- Placeholder — content will be generated by the docs-writer agent -->
```

### 5. Write scaffold manifest

Write a manifest to `<base-path>/scaffold/scaffold-info.md`:

```markdown
# Scaffold Manifest

**Workflow ID**: <id>
**Format**: mkdocs
**Output location**: <output-location>
**Date**: <YYYY-MM-DD>

## Files created

- `mkdocs.yml` — MkDocs configuration with nav structure
- `docs/index.md` — Landing page
- `docs/<category>/index.md` — Category landing pages (one per category)
- `docs/<category>/<topic>.md` — Placeholder module files (one per parent topic)

## Navigation structure

<Copy of the nav section from mkdocs.yml for reference>

## Categories

| Category | Directory | Topics |
|---|---|---|
| <name> | `docs/<category>/` | N topics |
```

### 6. Verify output

Verify that `mkdocs.yml` and `docs/index.md` exist at the output location.
